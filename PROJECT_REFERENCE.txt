--------------------------------------------------------------------------------
INTRO
--------------------------------------------------------------------------------

SentinelX is documented here for technical and industry readers: what is in the
repo, how parts connect, and what is fully wired vs scaffolded.

Real-time SOC-style updates use WebSockets (browser WebSocket API). The server
pushes JSON when logs are ingested, threats scored, alerts/incidents created,
and SOAR actions simulated. REST is used for triggers and health. See
sentinelx/README.md for the burst test (POST /simulate/burst).

2. PROJECT OVERVIEW
================================================================================

2.1 PROBLEM SPACE
-----------------
Enterprise-style threat intel / SIEM / SOAR scaffold: alerts, incidents, logs,
detection views, settings, simulated automation.

Live updates: push over WebSockets so multiple tabs get the same events without
polling. REST triggers work; WebSockets stream results.


2.2 LAYERS
----------
  1) Next.js UI — landing, auth, SOC shell; WebSocket hooks where enabled
  2) detection-engine — REST ingest + rule detect + score + broadcast WS
  3) backend FastAPI — health, docs, /ws hub, v1 auth only (wired)
  4) Data — Supabase (schema + engine writes) OR Compose Postgres/Redis


2.3 WEBSOCKET CHANNELS (DETECTION ENGINE)
-----------------------------------------
Each URL path has its own subscriber set (ConnectionManager).

  /ws/threats
    Payload types: log, threat, system (welcome on connect)
    Log lines and threat objects as produced.

  /ws/alerts
    Payload types: alert, incident, soar_action, system
    Alerts, incidents, simulated SOAR actions.

SOAR simulations emit type soar_action on the alerts channel.
Many client hooks open BOTH sockets.


2.4 DATA FLOW (TEXT DIAGRAM — replaces Mermaid for .txt sharing)
-----------------------------------------------------------------
  Browser (Next.js)
       |-- React Query / Zustand / Supabase clients
       |-- REST ------> detection-engine REST (ingest, health, etc.)
       |-- REST ------> backend /api/v1 (auth, etc.)
       |-- WebSocket -> detection-engine /ws/threats and /ws/alerts
       |-- WebSocket -> backend /ws (optional alternate hub)

  detection-engine: REST -> asyncio queue -> worker -> detect/score ->
                    broadcast to WS channels -> optional Supabase inserts

  Supabase: optional cloud/project using schema.sql


2.5 INGEST PATH (SUMMARY)
-------------------------
  1) POST /ingest-log (IngestLogPayload)
  2) Optional x-api-key vs SENTINELX_INGEST_API_KEY
  3) Queued (max 10000), HTTP returns accepted immediately
  4) Worker: detect, broadcast log on threats channel, Supabase + audit if on
  5) If findings: threat on threats; alert if score >= 75 (600s dedup);
     incident if score >= 90 or account_takeover; soar_action on alerts
        (e.g. block_ip if score >= 80)

Broadcast = push to every socket on that channel (multi-tab sync).

================================================================================
4. INDUSTRY SUMMARY
================================================================================

SentinelX shows how to split analyst UI (Next.js) from telemetry processing
(FastAPI detection engine) with PostgreSQL-friendly tenancy/audit (Supabase
schema or self-hosted Postgres). WebSockets deliver sub-second multi-subscriber
updates.

Use cases: SOC prototypes, training, integration demos (HTTP ingest + WS fan-out),
reference architecture for UI vs engine with explicit push channel.

Scale notes: in-process queue and in-memory dedup are single-instance; WS fan-out
is O(n) per channel; production scale usually needs shared queues, pub/sub,
and gateway patterns. Phase language in sentinelx/README.md applies.


================================================================================
APPENDIX — REPO CAVEATS (FACTUAL)
================================================================================

  - Compose maps backend 8080:8080 but container listens 8000; healthcheck 8000.
  - Compose wants sentinelx/Dockerfile; repo has Dockerfile.production.
  - prometheus.yml references missing prometheus_rules.yml and node-exporter;
    backend scrape uses /health (JSON) not Prometheus format.
  - Grafana and frontend both use host port 3001 in compose (collision).
  - nginx limit_req zones: definitions vs zone=one/api/frontend may not match.
  - more_clear_headers may need nginx module not in stock alpine image.
  - nginx /ws proxies to backend, NOT to detection-engine; no detection-engine
    service in compose — point WS env vars at host :8001 or extend proxy.
  - numpy/pandas/sklearn not used in detection-engine/main.py.
